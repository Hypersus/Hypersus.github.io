golang的内存逃逸分析

Posted on 2022-09-18 In golang

前言

在讨论内存逃逸之前，我们需要区别两个内存里的概念，栈内存和堆内存。golang中涉及到的内存分区主要分为四部分，分别是代码区、数据区、堆区和栈区。（数据区和堆区的区别大概在于数据区中存储的是静态的全局变量或常量，数据区的讨论暂时按下，没有查找到太多的资料）

栈内存用于存储函数体中的局部变量、函数参数、函数返回值、函数调用的临时上下文等。每个goroutine通常占用的内存不到10kb，因此每个goroutine获得的栈区通常也不会超过10kb。分配给一个goroutine的栈内存空间通常是连续固定的，由高地址向低地址增长，因此对栈内存的访问会很快。

堆内存用于动态地分配内存空间，内存大小不固定，由低地址向高地址增长。通常由操作系统维护一个空闲内存的链表，申请堆内存时遍历链表找到空闲的内存块（通常找到最接近申请大小的最小内存块）然后分配内存，在时间效率上比栈内存要慢。

GC针对的是堆内存回收。

内存逃逸

内存逃逸指的是，本来应该分配在栈内存中的某些变量或对象，逃逸到了堆内存上。

四种典型场景

局部指针返回

栈空间不足

当goroutine中的局部变量请求的内存空间大小比一个goroutine所能分配到的栈内存空间大时，变量会从栈内存逃逸到堆内存上。

动态类型

最简单和最常用的一个函数fmt.Println函数，由于入参是...interface{}，同时包括了接口和不定参数，也会发生内存逃逸。

golang中的闭包

Posted on 2022-08-23 Edited on 2022-08-24 In golang

在A Tour of Go中写到，A closure is a function value that references variables from outside its body，即闭包是父函数中的一个子函数，通常作为return值返回给调用者，在闭包中可以调用父函数中的局部变量并进行修改，父函数调用返回后，该局部变量因为在闭包中被引用，所以不会被GC回收而是会长久的存在于内存中。

由此可以推得，在go中变量的生命周期不由它的作用域决定。

golang中切片作为函数参数

Posted on 2022-08-19 In golang

回顾下slice的结构，gopl中写到，一个slice由三个部分构成：指针、长度和容量。当把一个slice作为函数参数进行传递时，实际上是把该指针进行传递。

在被调用函数（Callee）中的slice是该指针的一个副本，在被调用函数中对副本进行任何的修改都不会影响到调用者函数（Caller）中的slice结构（Callee中append的元素将不会被Caller访问到）

如果修改初始化stack时的capacity为0或1，则发现Caller和Callee中的stack指针地址不同，因为在Callee中进行了数组扩容，切片指向的底层数组已经有所不同。

Docker容器中的timezone问题

Posted on 2022-08-04 Edited on 2022-08-05 In Docker

前言

写完一段golang后写了Dockerfile把程序部署到Docker容器中，由于程序中涉及到日志打印（使用log.Printf打印日志），发现日志打印的时间是UTC时区，而我所在的时区是CST时区，查阅资料过程中提取到两个关键词$TW、/etc/localtime，两个共同作用决定终端下date命令输出代表的时区。

终端下的时区设置

当环境变量$TW没有设置时，date按照/etc/localtime指定的时区输出，通过file /etc/localtime可以看到这是一个软链接。

root权限下，$TW环境变量没有设置。

$TW变量可以设置为以/usr/share/zoneinfo为相对路径的路径例如TW=America/Los_Angeles、TW=Aisa/Shanghai

Docker容器中的时区设置

容器的时区设置和终端下的时区设置类似，默认时区由基础镜像决定，即Dockerfile中的FROM指定的镜像，golang镜像的默认时区是UTC0

可以在Dockerfile中通过ENV设置环境变量或者docker run -e $TW=Asia/Shanghai来设置时区

常量

  • 常量不同于变量的在运行期分配内存，常量通常会被编译器在预处理阶段直接展开，作为指令数据使用，所以常量是无法取地址的

  • golang的常量是一种在源码编译期间被创建的语法元素。它的初始化表达式必须是在编译期间可以求出值来的。而且，Go 常量一旦声明并被初始化后，它的值在整个程序的生命周期内便保持不变。这样，我们在并发设计时就不用考虑常量访问的同步。

  • golang对类型安全有着严格要求，即使两个类型具有相同的底层类型，但他们仍然是不同的数据类型，不可以被相互比较或混在一个表达式中进行运算。对于有类型常量（Typed Constant）需要进行显式的类型转换；对于无类型常量（Untyped Constant），再进行相关运算时编译器会进行隐式的类型转换。
