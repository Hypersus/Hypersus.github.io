关于CMD

Dockerfile中CMD的书写有三种形式

  • CMD ["executable","param1","param2"](exec form and a preferred form)
  • CMD ["param1","param2"]
  • CMD command param1 param2(shell form)

需要注意的一点是一个Dockerfile中只可以存在一条CMD指令。

CMD存在的作用是提供docker run的缺省值，在Dockerfile中没有ENTRYPOINT指令，且docker run [options] <image>没有指定后续的命令时，通过docker run运行的容器会运行Dockerfile中由CMD指定的命令。

关于ENTRYPOINT

有两种书写形式

  • ENTRYPOINT ["executable", "param1", "param2"]（exec form）
  • ENTRYPOINT command param1 param2（shell form）

Docker中的storage driver

Posted on 2022-07-20 Edited on 2022-07-21 In Docker

Docker使用storage driver来存储镜像，storage driver的写速度低于原生文件系统，尤其对于使用写时复制（copy on write）的文件系统，所以当应用包含很多写操作时（如数据库存储），应尽量使用Docker volume（即利用原生的文件系统）。

镜像和层

Docker镜像由一系列的层组成，每一层对应着Dockerfile中的一行指令，除了最后一层（即容器层）其他层都是只读的。

只有当进行文件增减操作时，Docker才会创建一个新的层（存疑，按下）

容器和层

容器和镜像的区别是，容器在镜像文件的只读层（read-only layer）上增加了一个可写层（writable layer），所有在容器中进行的文件增减操作都会记录在这个临时的可写层上。基于同一个镜像的容器100%共享所有的只读层

容器在硬盘上占用空间大小

使用docker ps -s打印容器占用空间大小信息，给出的信息包括size和virtual size

  • size: 该容器的可写层（writable layer）在硬盘上的占用空间大小
  • virtual size: 该容器在硬盘上占用的空间大小（包括可写层和属于镜像的只读层）

由于共享镜像的不同容器实际上在硬盘上只有一个只读镜像文件，因此所有容器在硬盘上占用的空间大小并不只是简单地把各个容器的virtual memory相加

Copy-on-Write (CoW)策略

CoW策略通过共享和复制来提高效率，具体来说，当某一层需要读取前一层已经存在的文件或目录时，该层可以直接读取已经存在的文件或目录，而当该层需要修改这个文件或目录时（执行写操作write），则对这个文件或目录复制到这一层再进行修改，这可以最小化I/O操作的开销，减小后续层的大小。

共享镜像中的层

使用docker pull拉取镜像后存放在/var/lib/docker/<storage-driver>目录下，每一个文件夹对应镜像中的一层（文件夹的名字和镜像的层ID不相同），CenOS7.7上该路径为/var/lib/docker/overlay2

使用docker image history <image>查看该image的构建过程

使用docker image inspect <image>查看该image每一层的sha256后的id

Dockerfile用法

docker build命令读取Dockerfile并使用Dockerfile中的指令构建docker镜像，Dockerfile中每一个指令对应docker镜像中的一层。通过docker image history image_id查看。

FROM和ARG

Dockerfile中的FROM指令必须在最前面，只有ARG指令可以放在FROM之前，ARG指令用于在build stage外设置变量便于后续书写，语法为ARG VAR_NAME=value，需要注意的一点是，如果要在除了FROM的其他指令中使用该变量，则需要在FROM后增加一句ARG VAR_NAME

RUN

有两种书写形式

  • RUN <command>
  • RUN ["executable","param1","param2"] (JSON array)

RUN指令执行对应命令并且在当前层上创建一个新层。

在docker中创建一个新层的开销是很小的（commits in docker is cheap），并且考虑到容器可以镜像中的任意一层为入口进行创建，因此按层堆叠RUN指令（把不同效果的RUN指令写在不同行）可以实现容器的source control

CMD

有三种书写形式

  • CMD ["executable","param1","param2"](exec form and a preferred form)
  • CMD ["param1","param2"]
  • CMD command param1 param2(shell form)

在一个Dockerfile中只可以存在一条CMD指令（但可以存在多条RUN指令）

CMD指令的主要作用是为容器提供defaults（docker run缺省要运行的命令时默认运行的命令），当在CMD中忽略executable时，必须使用ENTRYPOINT指令指定要运行的命令，在这种缺省executable的情况下，必须使用第二种书写形式（其余两种书写形式都包含了executable）

当用户使用docker run指定要运行的命令时，Dockerfile中由CMD指定的命令会。

ENTRYPOINT

有两种书写形式

  • ENTRYPOINT ["executable", "param1", "param2"]（exec form）
  • ENTRYPOINT command param1 param2（shell form）

:one:当使用exec form的ENTRYPOINT时，所有跟在docker run <image>后面的参数（即[command] [arg...]）会被添加到ENTRYPOINT的命令之后作为该命令的参数执行且CMD第二形式指定的参数将不再起作用。使用docker run --entrypoint可以覆盖ENTRYPOINT指令。

SRE vs DevOps

Posted on 2022-07-15 In Linux

参考资料：TechWorld with Nana

从整体的概念上讲，DevOps的出现是为了加速软件从开发到部署，从二次开发到持续集成持续部署的流程，是一种在实际生产环境中让开发和运维相互协作的理念，探讨的是设计怎样的一个自动化部署流程内容可以促进协作。SRE是遵循DevOps的准则进行实际实现的一个独立工作岗位。

近年来DevOps也有独立出来成为一个专门的职位，相对于SRE来说，DevOps工程师更注重于软件部署交付的速度，而SRE工程师则更注重于部署的可靠性。

想不出什么奇技淫巧，就只能无脑遍历，假设数组长度为n，总共有(\(2^n-1\))种可能的非空子集，遍历所有非空子集需要O(\(2^n\))的时间复杂度。因为数组的每一位对于某个非空子集只可能有两种状态，在子集内和不在子集内，所以可以用n比特来表示对应下标的元素是否在该子集内。又因为n比特的子集可以是n-1比特的子集和第n个元素或运算的结果，因此可以采用递归。我们设空子集的或运算结果为0，则在遍历时不需要特地判断空子集的情况。
